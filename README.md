# NSEdge-Validation

Reproducibility and benchmarking harness for **NSEdge** — an [ns-3](https://www.nsnam.org/)
contrib module (`ns3-mcs`) that simulates the **Device → Edge → Fog → Cloud** continuum at
**packet-level fidelity** (real TCP/IP stack, optional 802.11 Wi-Fi PHY/MAC) with
**profiling-driven compute** (per-task execution time drawn from measured hardware profiles).

This repository does two things:

1. **Physical validation** (`validation/`) — calibrates NSEdge against a physical testbed
   (loopback + `tc netem`) on RTT, packet-loss SLA reliability, and scale-out behaviour.
2. **Cross-simulator benchmark** (`benchmark/`) — measures NSEdge's speed and fidelity against
   three other edge simulators (EdgeSimPy, EdgeCloudSim, Simu5G) on a common workload.

> **Measurement policy.** Every number in `results/` is produced by actually executing the
> corresponding simulator and parsing its native output under `/usr/bin/time`. No delay,
> wall-clock, or reliability figure is hardcoded or computed from an analytical proxy. Where a
> simulator *cannot* express a quantity (e.g. flow-level engines have no packet loss), we report
> that as a limitation rather than substituting a fabricated value.

## Repository layout

```text
NSEdge-Validation/
├── README.md                 # this file
├── Dockerfile                # container for the Python/analysis environment
├── configs/
│   ├── dipdce_benchmark.json     # single source of truth for the cross-sim benchmark
│   └── profiles_validation.json  # measured hardware compute profiles (Intel-i5)

│   ├── measure_nsedge_wifi.py    # NSEdge (802.11ac)  -> results/nsedge_wifi_measured.json
│   ├── measure_edgesimpy.py      # EdgeSimPy (flow)   -> results/edgesimpy_measured.json
│   ├── measure_edgecloudsim.sh   # EdgeCloudSim (flow)-> results/ecs_measured.json
│   ├── measure_simu5g.sh         # Simu5G (5G-NR)     -> results/simu5g_measured.json

│   ├── worker.py                 # physical edge-worker process (loopback baseline)
│   ├── orchestrator.py           # offload orchestrator
│   ├── profiler.py               # hardware compute profiler
│   ├── extensive_validation.py   # RTT / loss / jitter sweeps vs. netem
│   ├── fl_validation.py          # flow-level (many-node) scaling
│   ├── scale_test.py             # scale-out sweeps
│   ├── deepdecision_*.py         # DeepDecision baseline comparison
│   └── journal_plots/            # generate_data.py + plot_data.py (+ data/)
├── experiments/              # Individual experiment environments
│   ├── 01_baseline_benchmark/
│   ├── 02_service_migration/
│   └── 03_wireless_benchmarks/
```

NSEdge itself lives in the sibling ns-3 tree: `../ns-3/contrib/ns3-mcs/`, with the benchmark
scenario at `../ns-3/scratch/dipdce-benchmark.cc`.

## Installation

The benchmark needs four independent toolchains. `install/bootstrap_sims.sh` recreates the
three external ones from scratch (they are intentionally **not** vendored):

```bash
# 1. ns-3 + NSEdge (once):  
cd ../ns-3 && ./ns3 build
cd ../NSEdge-Validation

# 2. Everything else:
bash install/bootstrap_sims.sh
```

## Step-by-Step Running Guide

To run the experiments and generate the plots, see the dedicated READMEs inside each specific experiment folder located in `experiments/`:

1. **[01_baseline_benchmark](experiments/01_baseline_benchmark/README.md):** Learn how to run the multi-simulator baseline benchmarks evaluating NSEdge against EdgeSimPy, EdgeCloudSim, and Simu5G, and generate the delay, reliability, and speed comparison plots (`delay_accuracy.pdf`, `speed_comparison.pdf`).
2. **[02_service_migration](experiments/02_service_migration/README.md):** Run the high-fidelity pre-copy live migration workload to validate TCP slow start, congestion avoidance, and bandwidth constraints compared directly against a physical emulation testbed, producing `migration_response_time.png`.
3. **[03_wireless_benchmarks](experiments/03_wireless_benchmarks/README.md):** Rerun the core benchmarks natively using Wi-Fi 802.11ac MAC-layer simulation and 5G-NR sub-6 GHz links in ns-3.

All generated metrics will be deposited into the root `results/` folder, and generated graphs into the root `figs/` directory by default.
