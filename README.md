# NSEdge-Validation

Reproducibility and benchmarking harness for **NSEdge** — an [ns-3](https://www.nsnam.org/)
contrib module (`ns3-mcs`) that simulates the **Device → Edge → Fog → Cloud** continuum at
**packet-level fidelity** (real TCP/IP stack, optional 802.11 Wi-Fi PHY/MAC) with
**profiling-driven compute** (per-task execution time drawn from measured hardware profiles).

This repository does two things:

1. **Physical validation** (`validation/`) — calibrates NSEdge against a physical testbed
   (loopback + `tc netem`) on RTT, packet-loss SLA reliability, and scale-out behaviour.
2. **Cross-simulator benchmark** (`benchmark/`) — measures NSEdge's speed and fidelity against
   three other edge simulators (EdgeSimPy, EdgeCloudSim, Simu5G) on a common workload.

> **Measurement policy.** Every number in `results/` is produced by actually executing the
> corresponding simulator and parsing its native output under `/usr/bin/time`. No delay,
> wall-clock, or reliability figure is hardcoded or computed from an analytical proxy. Where a
> simulator *cannot* express a quantity (e.g. flow-level engines have no packet loss), we report
> that as a limitation rather than substituting a fabricated value.

## Installation

The benchmark needs four independent toolchains. `install/bootstrap_sims.sh` recreates the
three external ones from scratch (they are intentionally **not** vendored):

```bash
# 1. ns-3 + NSEdge (once):  cd ../ns-3 && ./ns3 build
# 2. Everything else:
bash install/bootstrap_sims.sh      # ~15 min: OMNeT++ 6.0.3, INET 4.5.0, Simu5G,
                                    #          conda envs sim_env + edgesimpy_env, JDK 17
```

See `install/install.md` for the detailed, host-specific steps and prerequisites
(bison/flex/cmake, a C++17 compiler, `wget`). Build status is logged to
`third_party/rebuild_status.txt`.

## Quickstart — reproduce the benchmark

The benchmark is defined entirely by `configs/dipdce_benchmark.json` (workload = the DipDCE
image-offload architecture: 180 KB inference frames, 5 fps per edge worker, 132.44 ms measured
compute per frame, injected identically into every engine).

```bash
PY=third_party/micromamba_root/envs/sim_env/bin/python
ESP=third_party/micromamba_root/envs/edgesimpy_env/bin/python

# Measure each engine (writes results/<engine>_measured.json):
$PY  benchmark/measure_nsedge.py            # ~15 min (incl. 64-worker scale point)
$ESP benchmark/measure_edgesimpy.py         # < 1 min
bash benchmark/measure_edgecloudsim.sh 3    # averages 3 runs
bash benchmark/measure_simu5g.sh 10         # 10 s of 5G-NR

# Consolidate + render the four figures + the LaTeX table:
$PY  benchmark/plot_benchmark.py
#   -> results/benchmark_metrics.json, results/benchmark_table.tex, figs/fig_*.pdf
```

### Measured results (4 edge workers, 180 KB frames, injected 132.44 ms compute)

| Simulator     | Model | Real-time factor | Delay (ms) | SLA @3% loss | Models loss? | Peak RSS |
|---------------|-------|-----------------:|-----------:|-------------:|:------------:|---------:|
| **NSEdge**    | packet (Ethernet)  | **2.0**            | 162        | **51%**      | ✅ | 65 MB  |
| EdgeSimPy     | flow               | 3.6×10³            | 135        | 100%         | ❌ | 87 MB  |
| EdgeCloudSim  | flow (CloudSim)    | 309                | 414†       | 100%         | ❌ | 134 MB |
| Simu5G        | packet (5G-NR)     | 2.7                | 293‡       | —            | ✅ | 99 MB  |

*Physical baseline: RTT 143.6 ms, SLA reliability 67.2% under 3% loss.
†native CloudSim service time (different compute basis). ‡per-packet 5G-NR transport
(different access technology).*

**Headline:** NSEdge runs *faster than real time* while computing full packet dynamics, and is
the cheap-enough engine that **uniquely reproduces the SLA collapse under loss** (96% → 51%,
tracking the physical 67.2%). Flow-level engines are ~1000× faster but structurally cannot model
a lost packet, so they report a spurious 100%.

## How to experiment with different things

Almost everything is driven by the two JSON configs — no code edits needed for the common cases.

| I want to… | Do this |
|---|---|
| **Change the workload** (frame size, fps, workers, duration, SLA) | edit `configs/dipdce_benchmark.json` → re-run the `measure_*` scripts. All engines read this file. |
| **Change the scalability sweep** | edit `scalability_sweep.edge_workers` in the same config. |
| **Add a new hardware profile** (e.g. a GPU) | add a `hardware_profiles[]` entry to `configs/profiles_validation.json` (fields: `input_bytes`, `execution_time_ms{mean,stddev}`, power, util) and pass `--profile`/`--hw_id` to the NSEdge scenario. This unlocks the real 30 fps DipDCE regime (see TODO). |
| **Sweep packet loss / jitter** | NSEdge: `measure_nsedge.py` already runs a 3% loss point; change `loss_test.loss_rate`, or call the scenario directly with `--loss` / `--jitter_us`. |
| **Use a different access technology** | the NSEdge scenario supports `NetworkTech::{ETHERNET_1G, WIFI_80211AC/AX, NR_5G_SUB6, …}`; `measure_nsedge_wifi.py` shows the Wi-Fi path. |
| **Change the orchestration policy** | subclass `OrchestrationPolicy` in the scenario (see `DipDceEdgePlacement` in `dipdce-benchmark.cc`) and register it with `JobOrchestrator`. |
| **Add another simulator to the comparison** | add a `benchmark/measure_<engine>.{py,sh}` that emits the same JSON schema (list of `{scenario, workers, wall_s|sim_wall_s, mean_delay_ms, sla_pct, rss_kb, …}`) and add it to the engine list in `plot_benchmark.py`. |
| **Re-run the physical validation** | see `validation/extensive_validation.py` (RTT/loss/jitter) and `validation/scale_test.py`; figures via `validation/journal_plots/`. |

Run the NSEdge benchmark scenario directly for ad-hoc experiments:

```bash
cd ../ns-3 && export LD_LIBRARY_PATH=$PWD/build/lib:$LD_LIBRARY_PATH
./build/scratch/ns3.43-dipdce-benchmark \
    --duration=30 --lambda=20 --workload_size=180000 --num_nodes=4 \
    --bandwidth=1Gbps --latency_ms=0.39 --sla_ms=250 --loss=0.0 \
    --profile=../NSEdge-Validation/configs/profiles_validation.json --outdir=results-bench
```

## TODO — experiments & validation needed for a strong journal paper

Ordered roughly by impact on reviewer confidence. Items marked **(blocker)** are, in our view,
prerequisites for a top-tier SimPat submission.

### Fidelity & ground truth
- [ ] **(blocker) GPU compute profiling.** All results use a measured Intel-i5 *CPU* profile
      (132 ms/frame → ~5 fps/worker). DipDCE's headline 30 fps regime needs a measured
      GPU/accelerator profile (e.g. GTX-1080Ti/YOLOv8). Add it to `profiles_validation.json`
      and re-validate; this is currently the single biggest gap.
- [ ] **(blocker) Multi-node physical testbed.** The baseline is loopback shaped with
      `tc netem`, not real hardware. Reproduce RTT/loss/SLA on a small real cluster (≥4 nodes,
      real NICs/switch) to make the fidelity claim robust to hardware effects.
- [ ] **Wireless ground truth.** NSEdge has a real 802.11 PHY/MAC path (`YansWifi` +
      `AdhocWifiMac`); validate the Wi-Fi 802.11ac delay/contention numbers against a physical
      Wi-Fi link, and add a 5G-NR validation against Simu5G or a testbed.
- [ ] **Jitter on wired links.** The current model applies delay jitter on the analytical
      fallback path, not the socket path (a stated limitation). Implement per-packet wired
      jitter and re-run the `net_phys_jitter` comparison.

### Statistical rigor
- [ ] **(blocker) Confidence intervals on the cross-sim benchmark.** Report mean ± 95% CI over
      ≥5 seeds for every engine/metric; add a significance test for the loss-reproduction claim.
- [ ] **Matched arrival processes.** NSEdge shows ~29 ms edge-queueing at the operating point
      while the deterministic EdgeSimPy driver shows none — align the arrival process (Poisson
      vs. periodic) across engines so the delay axis is strictly apples-to-apples.
- [ ] **Cross-engine loss sweep.** Extend the loss experiment to Simu5G (via `targetBler`) and a
      sweep of loss rates {1,3,5,10}% to show NSEdge and Simu5G both track physical degradation
      while flow-level engines stay flat at 100%.

### Coverage / breadth
- [ ] **Scale-out to large topologies.** Push the scalability curve to 100–1000 nodes using the
      flow-level fast path; characterise where packet-level NSEdge becomes impractical (it is
      super-linear) and where the flow model suffices.
- [ ] **Mobility & handover.** NSEdge ships a mobility manager; add a mobility scenario (UE
      movement, edge handover) and validate migration/latency behaviour — a common SimPat theme.
- [ ] **Service migration.** Exercise and validate live-migration cost across the continuum.
- [ ] **Energy validation.** NSEdge models compute + network energy; validate against measured
      power for at least one workload.
- [ ] **A second workload.** Generalise beyond DipDCE (e.g. video analytics pipeline, federated
      learning round, AR/VR frame) to show the framework is not overfit to one architecture.

### Reproducibility / engineering
- [ ] **One-command reproduction.** Wrap `install/bootstrap_sims.sh` + the `measure_*` +
      `plot_benchmark.py` into a single `make reproduce` (or the `Dockerfile`) with pinned
      versions and fixed seeds.
- [ ] **CI smoke test.** A short (1–2 min) end-to-end run on every commit to catch pipeline rot.
- [ ] **Fairness disclosure.** Document, in the paper, exactly what compute is injected vs.
      native per engine (already summarised in `docs/methodology.md`).

## Notes & honest limitations

- **CPU-only compute** (see TODO). GPU profiles are future work; the *network* path is the
  object of the cross-simulator comparison, so compute is injected identically across engines.
- **Delay is not directly comparable across access technologies.** Simu5G models 5G-NR and
  EdgeCloudSim uses its own service model; only NSEdge and EdgeSimPy are compared on absolute
  delay (shared wired access + identical injected compute). See `docs/methodology.md`.
- **FogNetSim++ is excluded** — it requires OMNeT++ 4.6 and does not build on the current
  toolchain. It is documented, not proxied.
