# Service Migration Experiment (Follow Me at the Edge)

This directory isolates all code, data, and plots for the service migration validation.
The experiment models the "Follow Me at the Edge" (Ouyang et al.) policy, involving a 1 GB state transfer via pre-copy live migration. 

It compares the performance of the proposed **NSEdge** simulator against a **physical emulation testbed** (using Linux `tc netem`).

## Contents
- `results/`: Contains the metrics (`ns3_tasks.csv`, `physical_tasks.csv`) generated by the simulator and physical testbed.
- `figs/`: Contains the generated response time validation plots.
- `plot_migration.py`: Python script used to parse the CSV files and generate the comparative scatter plot.
- `run_migration_physical.sh`: Shell script to execute the physical emulation testbed.

## 1. Running the NS-3 Simulation (NSEdge)
The NS-3 simulation models the migration policy using realistically sized buffers and TCP windowing. 
To ensure the simulation aligns with realistic Linux TCP behaviors, the `ns3-migration-scenario.cc` script explicitly increases `TcpSocket::SndBufSize` and `RcvBufSize` to 4 MB to prevent artificial sender-side bufferbloat when the RTT increases.

**Execution (Ethernet Baseline):**
```bash
cd /proj/oasees-PG0/NS3-Edge/ns-3
./ns3 run scratch/ns3-migration-scenario
```
*(The simulation automatically saves its output tasks to `results-migration/tasks.csv`)*

**Execution (5G-NR):**
You can also run the migration scenario utilizing a 5G-NR sub-6 GHz link to demonstrate the impact of 5G transport characteristics on the migration performance.
```bash
cd /proj/oasees-PG0/NS3-Edge/ns-3
./ns3 run 'scratch/ns3-migration-scenario --tech=5g'
```

## 2. Running the Physical Emulation Testbed
The physical testbed runs two edge worker processes and a state receiver locally, shaping the `lo` loopback interface using `tc qdisc` and `htb` to mirror the delay/bandwidth conditions of the NS-3 simulation.

**Critical Emulation Details:**
1. **Symmetric Delay (`tc`)**: We use `tc qdisc change` to dynamically adjust delay without dropping packets. Furthermore, we apply `tc filter` rules for **both** `dport` and `sport` (e.g., 8888, 8889). This guarantees that both the outgoing data payloads and the returning TCP ACKs pass through the delay queue, perfectly simulating a symmetric RTT.
2. **TCP Metrics Caching (`sysctl`)**: Because the testbed orchestrator establishes multiple sequential TCP connections to `127.0.0.1`, Linux will aggressively cache the TCP congestion window (`cwnd`). This accidentally bypasses TCP Slow Start for new connections, resulting in artificially flat response times even as RTT increases. We strictly **disable** TCP metric caching (`net.ipv4.tcp_no_metrics_save=1`) and flush the metrics before running the testbed so that realistic multi-RTT Slow Start phases are observed for the 180 KB payloads.

**Execution:**
```bash
cd /proj/oasees-PG0/NS3-Edge/NSEdge-Validation/validation
./run_all.sh
```
*(The `run_all.sh` script applies the sysctl rules, launches the workers, runs the `migration_orchestrator.py` script, and outputs `tasks.csv`)*

After completion, move the results to this experiment directory:
```bash
cp /proj/oasees-PG0/NS3-Edge/NSEdge-Validation/validation/tasks.csv /proj/oasees-PG0/NS3-Edge/NSEdge-Validation/experiments/02_service_migration/results/physical_tasks.csv
```
